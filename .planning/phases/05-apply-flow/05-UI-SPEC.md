## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Checkbox-to-chevron gap, inline badge padding |
| sm | 8px | Compact spacing within footer bar, gap between action buttons |
| md | 16px | Default element spacing, banner internal padding, footer horizontal padding |
| lg | 24px | Footer bar horizontal padding, banner margin from edges |
| xl | 32px | Section gap (inherited from Phase 3) |
| 2xl | 48px | Space between parameter browser bottom and footer bar content area |
| 3xl | 64px | Not used in this phase |

Exceptions: Footer bar uses 12px vertical padding (`py-3`) for a compact persistent bar that does not consume excessive vertical space. This is an intentional deviation from the strict 4px multiple scale (12 = 3 * 4, so it remains grid-aligned).

## Color

Inherits Phase 3 color contract. Phase 5 adds these specific role assignments:

| Role | Token / Value | Usage |
|------|---------------|-------|

| Secondary (30%) | `bg-bg-tertiary` (#313a56) | Dimmed section overlay effect |
| Accent (10%) | `text-warm` (#c8a46a) | Apply button (primary variant), success flash |
| Warning | `text-status-warning` (#bea873) / `bg-status-warning` | Health check banner border and title |
| Success | `text-status-success` (#a3c48e) | Post-apply success indicator text |

### Accent Reserved For (Phase 5 additions)

- **Warm gold** (`action-button-primary`): Apply button only. This is the single primary CTA in the footer.
- **Status warning** (`text-status-warning`, `border-status-warning/30`, `bg-status-warning/[0.08]`): Health check warning banner border, icon, and title text.
- **Status success** (`text-status-success`): Post-apply success indicator text flash.
- **Accent violet** (`accent-accent`): Checkbox accent color for the native `<input type="checkbox">` elements.

### Dimmed Section Treatment (D-03)

Unchecked (not accepted) sections receive `opacity-[0.45]` on their entire `ParameterSection` container. The section remains interactive (checkbox and collapse still work) but visually communicates "these changes will not apply." The 0.45 value is at the lower end of the D-03 range (40-50%) to create clear visual separation between accepted and not-accepted sections.

Source: CONTEXT.md D-03, index.css status tokens, Phase 3 color contract.

## Focal Point

**Primary visual anchor:** The sticky footer bar. In Phase 5 the footer bar is always visible and serves as the persistent decision point. The dynamic "Apply N Sections" label, combined with the section count summary, gives builders constant awareness of their selection state. The health check warning banner, when present, becomes a secondary focal point drawing attention through its amber border and icon.

## Component Inventory

### 1. Section Checkbox (modification to ParameterSection)

- **Position**: Inserted before the chevron in the existing ParameterSection header button, only when `hasPreset` is true.
- **Element**: Native `<input type="checkbox">` with Tailwind styling.
- **Size**: `h-4 w-4` (16x16px). Matches the chevron's visual weight.
- **Color**: `accent-accent` (#a897d2) for the checked state fill. Unchecked shows default dark border.
- **Click behavior**: `e.stopPropagation()` on the checkbox to prevent triggering the collapse toggle on the parent button.
- **Change handler**: Calls `toggleAccepted(section)` on the wizard store.
- **Default state**: All checkboxes checked when a preset is first selected (D-02).
- **Layout**: `flex items-center gap-3` on the parent row. Order: checkbox, chevron, section name, badges.

### 2. ApplyFooterBar (new component)

- **Position**: Sticky to the bottom of the TuningWizard scroll container. `sticky bottom-0 z-10`.
- **Visibility**: Only rendered when a preset is selected (`selectedPresetId !== null`).
- **Background**: `bg-bg-primary` with `border-t border-border-muted` top border. The solid background prevents parameter rows from showing through.
- **Layout**: Flex row with `items-center justify-between`. Left side: section count summary. Right side: action buttons row with `gap-3`.
- **Padding**: `px-6 py-3` (24px horizontal, 12px vertical).
- **Animation**: Enters with `animate-unfurl-in` (400ms ease) when preset is first selected.
- **Shadow**: `shadow-[0_-4px_16px_rgba(8,10,18,0.3)]` upward shadow to visually float above content.

#### Left Content: Section Count Summary

- **Text**: `"{N} of {total} sections selected"` in Crimson Pro 14px weight 400 `text-text-secondary`.
- **Dynamic**: Updates immediately when checkboxes change.
- **Examples**: "3 of 4 sections selected", "0 of 4 sections selected", "4 of 4 sections selected".

#### Right Content: Action Buttons

Three buttons in a flex row with `gap-3`:

1. **Reset** (`ActionButton variant="ghost"`):
   - Label: "Reset"
   - Always visible when footer is visible.
   - Calls `resetWizard()` on the store.
   - No confirmation dialog (D-07 -- Reset only clears wizard UI state, never touches saved config).

2. **Undo** (`ActionButton variant="secondary"`):
   - Label: "Undo"
   - Only visible when `undoAvailable` is true (after a successful apply).
   - Calls `undoApply()` on the store.
   - No confirmation dialog (single-level undo is predictable, D-05).

3. **Apply** (`ActionButton variant="primary"`):
   - Label: `"Apply {N} Section"` (singular) or `"Apply {N} Sections"` (plural). Always includes the count.
   - Disabled when `acceptedCount === 0`.
   - Disabled when apply is in progress (prevent double-click).
   - Calls `applyPreset()` on the store.
   - No confirmation dialog (D-06 -- the label itself communicates scope).
   - While applying: show `Spinner` inside the button, label changes to "Applying..."

### 3. HealthCheckBanner (new component)

- **Position**: Rendered inside TuningWizard, between MetricSectionCards and SearchFilterBar. Only appears after an apply with mixed sections that triggers warnings.
- **Layout**: Flex row with `items-start gap-3`. Rounded container with amber border and tinted background.
- **Background**: `bg-status-warning/[0.08]` (amber at 8% opacity).
- **Border**: `border border-status-warning/30` (amber at 30% opacity). `rounded-lg` (8px radius).
- **Margin**: `mx-6 mt-4` (24px horizontal, 16px top).
- **Padding**: `px-4 py-3` (16px horizontal, 12px vertical).
- **Icon**: Unicode warning triangle `&#9888;` in `text-status-warning`, aligned to top of first message line.
- **Title**: "Balance Warning" in Crimson Pro 14px weight 600 `text-status-warning`.
- **Messages**: Each `HealthWarning.message` rendered as a `<p>` in Crimson Pro 14px weight 400 `text-text-secondary`. Spacing: `mt-1` between messages.
- **Dismiss**: Close button (`&#10005;`) at top-right. `ml-auto text-text-muted hover:text-text-primary`. Click calls `setHealthWarnings([])` to clear.
- **Non-blocking**: Builder can dismiss and continue. Banner does not prevent any action (D-08).
- **Animation**: Enters with `animate-unfurl-in` (400ms ease).

### 4. Success Indicator (inline in footer bar)

- **Trigger**: After a successful apply, `applySuccess` is set to true in the store.
- **Display**: A brief text flash appended to the left side of the footer bar, after the section count. Text: "Applied!" in Crimson Pro 14px weight 600 `text-status-success`.
- **Animation**: Fade in with `animate-saved-flash` (300ms, existing keyframe in index.css). Auto-dismiss after 2000ms via `setTimeout` calling `clearApplySuccess()`.
- **Layout**: Inline next to section count with `ml-3` gap. Uses `opacity` transition for smooth appearance/disappearance.

### 5. Dimmed Section Wrapper (modification to TuningWizard)

- **Implementation**: In the TuningWizard section loop, wrap each `ParameterSection` in a `<div>` with conditional `opacity-[0.45]` class when the section is not in `acceptedSections`.
- **Transition**: `transition-opacity duration-300` for smooth dimming/brightening when checkboxes are toggled.
- **Interaction**: Dimmed sections remain fully interactive. Checkbox, collapse, hover effects all still work. The opacity reduction is purely visual communication (D-03).

### Destructive Actions

This phase has no destructive actions. Reset clears only wizard UI state and never touches saved config (D-07). Undo restores a known-good snapshot and always succeeds. Apply writes through the existing `saveProjectConfig()` path which handles YAML round-trip safely.

## Interaction Contracts

### Section Acceptance Flow

1. **Preset selected**: All 4 section checkboxes appear and are checked (D-02). Footer bar slides in from bottom.
2. **Uncheck a section**: Section dims to 0.45 opacity. Footer count updates ("2 of 4 sections selected"). Apply button label updates to "Apply 2 Sections".
3. **Uncheck all sections**: Apply button becomes disabled. Footer shows "0 of 4 sections selected".
4. **Recheck a section**: Section brightens back to full opacity. Footer and button update.
5. **Switch presets**: All checkboxes reset to checked (D-10). Prior section selections are discarded.

### Apply Flow

1. **Click Apply**: Button shows Spinner + "Applying..." label. Store takes config snapshot, builds partial from accepted sections, merges and writes to configStore, saves to YAML.
2. **Apply succeeds**: Spinner removed. "Applied!" success text flashes in footer for 2 seconds. Undo button appears. Comparison view updates to show new current values (applied values become "current"). Metric cards recompute. If mixed sections were accepted, health check runs and banner may appear.
3. **Apply fails**: Button returns to normal state. Error is logged to console (no user-facing error banner for v1 -- save failures are extremely rare in local filesystem operations).

### Undo Flow

1. **Click Undo**: configStore receives the stored snapshot. YAML file is saved. Undo button disappears. Health warnings clear. Comparison view updates to reflect restored values.
2. **Sequential applies**: Each apply overwrites the previous snapshot. Only the most recent apply can be undone (D-12).

### Reset Flow

1. **Click Reset**: Wizard returns to initial state -- preset cards all unselected, no checkboxes visible, no footer bar, no health banner, search cleared, filters reset to all active (D-07).
2. **After Reset**: Config values remain unchanged. Only wizard UI state is cleared. The `configSnapshot` and `undoAvailable` are also cleared since the wizard is returning to its starting state.

### Health Check Flow

1. **Trigger**: Runs automatically after apply when not all 4 sections were accepted (`acceptedSections.size < 4`).
2. **Display**: Banner appears between MetricSectionCards and SearchFilterBar with `animate-unfurl-in`.
3. **Dismiss**: Builder clicks the X button. Banner disappears. Does not reappear until the next apply.
4. **No warnings**: If health check finds no issues, no banner appears. Apply proceeds silently.

### Scroll Behavior

- Footer bar is sticky to bottom of the scroll container, always visible.
- Parameter browser bottom padding must account for footer bar height (~56px) to prevent the last parameter rows from being hidden behind the footer. Use `pb-20` (80px) on the parameter browser container when footer is visible.
- Health check banner scrolls with content (not sticky).

## State Contract

Extensions to `tuningWizardStore` (adds to Phase 3 state):

| State Field | Type | Default | Purpose |
|-------------|------|---------|---------|
| acceptedSections | `Set<TuningSection>` | all 4 sections | Which sections will be included in the apply |
| configSnapshot | `AppConfig \| null` | `null` | Pre-apply config state for single-level undo |
| undoAvailable | `boolean` | `false` | Whether Undo button is visible |
| healthWarnings | `HealthWarning[]` | `[]` | Active health check warnings to display |
| applySuccess | `boolean` | `false` | Drives the "Applied!" flash indicator |

New store actions:

| Action | Signature | Behavior |
|--------|-----------|----------|
| toggleAccepted | `(s: TuningSection) => void` | Add/remove section from acceptedSections set |
| applyPreset | `() => Promise<void>` | Snapshot, merge accepted sections, write to configStore, save YAML, run health check |
| undoApply | `() => Promise<void>` | Restore snapshot to configStore, save YAML, clear undo state |
| resetWizard | `() => void` | Clear all wizard state to initial values (D-07) |
| setHealthWarnings | `(w: HealthWarning[]) => void` | Set or clear health warnings |
| clearApplySuccess | `() => void` | Set applySuccess to false (called by timeout) |

Source: RESEARCH.md Pattern 1, CONTEXT.md D-01 through D-12.
